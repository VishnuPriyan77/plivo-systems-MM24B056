The Flaky Network: Submission Summary

The solution replaces the naive UDP pass-through with a compact forward-error-correction protocol. It is designed for live 20 ms frames where waiting for retransmission would usually cost too much time. The recommended grading delay is --delay_ms 98.

System Path

The harness source gives the sender one exact 160-byte payload every 20 ms. The sender is only allowed to communicate with the receiver through the hostile relay, and the receiver must deliver exact seq + payload frames to the harness player before each deadline.

Harness SourceEmits frame i at T0 + i*20ms.
SenderEncodes media and repair packets.
RelayDrops, delays, reorders, and duplicates UDP packets.
ReceiverBuffers, repairs, and emits each frame once.
Harness PlayerScores correctness and deadline arrival.

Protocol Design

Relay-facing packets are 161 bytes: one compact control byte plus the 160-byte payload. The top two bits of the control byte mark packet type, and the low 6 bits carry the frame sequence residue. The receiver combines that residue with T0 and current time to recover the absolute sequence number; the provided delays are far smaller than the 64-frame wrap window.

PacketPayloadPurpose
Data Frame bytes for sequence i Normal delivery path for every frame.
Adjacent XOR repair payload[i] XOR payload[i+1] for an even/odd pair Recovers either frame if exactly one of the pair's data packets is lost.
Diagonal XOR repair payload[i-3] XOR payload[i] for most odd i Adds another local equation so clustered losses are recoverable more often.

Bandwidth Strategy

Every frame is sent once. Each two-frame pair gets one adjacent repair packet, and most pairs also get a diagonal repair packet. To stay under the strict 2.0x byte cap, the sender skips the diagonal repair every 32nd pair. On 30-second runs this produced 479136 relay bytes for 1500 frames, or about 1.996x raw payload overhead.

Receiver Behavior

  • Store received data frames by absolute sequence number.
  • Store adjacent and diagonal XOR equations near each frame.
  • When an equation has one known frame and one missing frame, reconstruct the missing frame with XOR.
  • Run a few local repair passes because one recovered frame can unlock another equation.
  • Send each available frame to the player immediately, and never emit the same sequence twice.

Why Not Retransmission

Retransmission needs a receiver request and a sender reply, and both packets cross the same hostile relay. At low playout delays, that round trip is usually too late. Forward repair is sent before a loss is known, which spends bandwidth but keeps recovery one frame away instead of one network round trip away.

Observed Result

The final design passed the visible moderate profile at 98 ms across tested seeds, including seeds where 97 ms and 90 ms failed. The main remaining failure mode is losing or late-delivering too many related data and repair packets in the same local equation graph.
